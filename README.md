# Сервис асинхронной проверки списка URL

Fullstack-приложение: REST API для асинхронной проверки доступности списка URL
и веб-интерфейс для работы с ним.

- **Бэкенд:** Node.js + TypeScript, **NestJS**, хранение in-memory
- **Фронтенд:** TypeScript, **React** + **Redux Toolkit**, Vite
- **Инфраструктура:** Dockerfile для каждого сервиса + `docker-compose`

## Быстрый старт (Docker)

Требуется установленный Docker с плагином Compose.

```bash
docker compose up --build
```

После сборки откройте:

- **Фронтенд:** http://localhost:8080
- **API:** http://localhost:3000/api

Nginx во фронтенд-контейнере проксирует запросы `/api/*` на бэкенд-сервис,
поэтому фронтенду не нужен отдельный CORS-конфиг.

Остановить: `Ctrl+C`, затем `docker compose down`.

## Локальный запуск (без Docker)

Требуется **Node.js 20+** (разработка велась на Node 22).

### 1. Бэкенд

```bash
cd backend
npm install
npm run start:dev      # http://localhost:3000/api
```

### 2. Фронтенд (в отдельном терминале)

```bash
cd frontend
npm install
npm run dev            # http://localhost:5173
```

Dev-сервер Vite проксирует `/api` на `http://localhost:3000` (см. `vite.config.ts`),
так что оба сервиса работают вместе из коробки.

Фронтенд разделён на три слоя, как требует ТЗ:

- **глобальное состояние** (`store/`) — список заданий, активное задание, детали,
  состояния загрузки/ошибок.

## REST API

Базовый префикс — `/api`.

### `POST /api/jobs` — создать задание

Запрос:

```json
{ "urls": ["https://example.com", "https://github.com"] }
```

Ответ `201 Created`:

```json
{ "jobId": "b1c2d3e4-..." }
```

Обработка URL стартует в фоне; ответ не ждёт её завершения.
Некорректный ввод (пустой список, не-URL) → `400 Bad Request`.

### `GET /api/jobs` — список заданий

Ответ `200 OK` (новые — первыми):

```json
[
  {
    "id": "b1c2d3e4-...",
    "createdAt": "2026-07-20T10:00:00.000Z",
    "status": "in_progress",
    "stats": {
      "total": 2, "pending": 0, "inProgress": 1,
      "success": 1, "error": 0, "cancelled": 0, "processed": 1
    }
  }
]
```

### `GET /api/jobs/:id` — детальная информация

Ответ `200 OK` — задание + статус каждого URL:

```json
{
  "id": "b1c2d3e4-...",
  "createdAt": "2026-07-20T10:00:00.000Z",
  "status": "completed",
  "stats": { "total": 2, "success": 1, "error": 1, "processed": 2, "...": "..." },
  "urls": [
    {
      "id": "u1-...",
      "url": "https://example.com",
      "status": "success",
      "httpStatus": 200,
      "error": null,
      "startedAt": "2026-07-20T10:00:00.100Z",
      "finishedAt": "2026-07-20T10:00:03.400Z",
      "durationMs": 3300
    }
  ]
}
```

Неизвестный `id` → `404 Not Found`.

### `DELETE /api/jobs/:id` — отменить задание

Помечает задание как `cancelled` и прекращает обработку **ещё не начатых** URL.
Уже выполняющиеся HEAD-запросы доводятся до конца. Возвращает актуальные детали задания.
Операция идемпотентна: повторная отмена/отмена завершённого задания просто возвращает его состояние.

### Статусы

| Задание (`status`) | URL (`status`)                              |
| ------------------ | ------------------------------------------- |
| `pending`          | `pending`                                   |
| `in_progress`      | `in_progress`                               |
| `completed`        | `success` / `error`                         |
| `cancelled`        | `cancelled`                                 |
| `failed`           | —                                           |

## Логика обработки

Согласно ТЗ:

- для каждого URL выполняется HTTP **HEAD**-запрос (глобальный `fetch`, с таймаутом);
- перед сохранением результата — **искусственная задержка** (случайно, 0–10 с);
- не более **5 одновременных** HEAD-запросов на одно задание
  (пул конкурентности в `utils/concurrency.ts`);
- несколько заданий обрабатываются **параллельно** (каждое запускается независимо в фоне);
- успех — ответ со статусом 2xx; ошибка — 4xx/5xx или сетевая ошибка/таймаут
  (сообщение об ошибке сохраняется в поле `error`).

## Как проверить вручную

### Вариант A — через веб-интерфейс (рекомендуется)

1. Запустите проект: `docker compose up --build` (или локально — бэкенд и фронтенд, см. выше).
2. Откройте http://localhost:8080 (Docker) или http://localhost:5173 (локально).
3. В форме «Новая проверка» вставьте несколько URL, по одному в строке, например:
   ```
   https://github.com
   https://www.google.com
   https://httpstat.us/404
   https://this-domain-does-not-exist-xyz.invalid
   ```
4. Нажмите «Запустить проверку». Проверьте, что:
   - справа появилось активное задание со статусом «Обработка» и прогрессом «X из Y»;
   - таблица URL заполняется статусами (Успех/Ошибка), HTTP-кодами и временем;
   - прогресс и статистика обновляются сами (поллинг), список слева тоже обновляет счётчики ✓/✕;
   - обработка идёт пачками — одновременно не больше 5 URL «в работе»;
   - по завершении статус становится «Завершено», опрос прекращается.
5. Создайте второе задание с большим списком и во время обработки нажмите «Отменить задание» —
   статус станет «Отменено», не начатые URL получат статус «Отменён».
6. Переключайтесь между заданиями в списке — детали корректно меняются, «хвост» опроса
   прошлого задания не портит интерфейс.

### Вариант B — через API (curl)

```bash
# создать задание
curl -s -X POST http://localhost:3000/api/jobs \
  -H 'Content-Type: application/json' \
  -d '{"urls":["https://github.com","https://httpstat.us/500","https://nope.invalid"]}'
# → {"jobId":"..."}

# список заданий
curl -s http://localhost:3000/api/jobs

# детали (подставьте jobId)
curl -s http://localhost:3000/api/jobs/<jobId>

# отменить
curl -s -X DELETE http://localhost:3000/api/jobs/<jobId>

# проверка валидации → 400
curl -s -o /dev/null -w "%{http_code}\n" -X POST http://localhost:3000/api/jobs \
  -H 'Content-Type: application/json' -d '{"urls":[]}'
```

Что стоит увидеть в ответах: сразу после создания статус `pending`/`in_progress`; во время
обработки часть URL в `in_progress`, часть уже `success`/`error`; у каждого URL есть
`httpStatus`, `error`, `startedAt`, `finishedAt`, `durationMs`; после `DELETE` — статус задания
`cancelled`, а не начатые URL — `cancelled`.

### Вариант C — автотесты

```bash
cd backend && npm ci && npm test && npm run test:e2e   # 19 unit + 6 e2e
cd ../frontend && npm ci && npm test                   # 16 тестов
```

## Ключевые технические решения

- **Отмена (AbortController + проверка перед стартом URL).** Каждому заданию
  соответствует токен отмены. Пул проверяет `signal.aborted` перед началом
  обработки очередного URL: не начатые помечаются `cancelled`, начатые доводятся
  до конца — ровно как в ТЗ. После завершения пула статус не «перетирается»
  в `completed`, если задание уже отменено.

- **Ограничение конкурентности без внешних зависимостей.** Небольшой пул воркеров
  (`runWithConcurrency`) держит не более N активных задач и берёт следующий URL
  по мере освобождения — прозрачно и покрыто тестом.

- **Корректный опрос на фронтенде.** Хук `useActiveJobPolling` опрашивает активное
  задание, пока статус не конечный; при смене активного задания старый интервал
  очищается. Дополнительно редьюсер **отбрасывает ответы по неактивному `jobId`**,
  поэтому «догоняющие» ответы прошлого задания не меняют интерфейс.

- **Живое обновление списка.** Пока в списке есть незавершённые задания, он опрашивается
  каждые 2 c (`useJobsListPolling`), поэтому статистика ✓/✕ в карточках обновляется в реальном
  времени; как только все задания в конечном статусе — опрос списка прекращается.

- **Разделение слоёв на фронтенде.** API-слой, Redux-стор и компоненты разделены;
  компоненты не обращаются к `fetch` напрямую.

- **In-memory хранилище за интерфейсом репозитория.** `JobsRepository` инкапсулирует
  `Map`, что позволяет при необходимости заменить хранилище на БД без изменений в сервисе.
